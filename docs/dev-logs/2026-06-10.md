## Round B — Swan Lake "from $80" mismatch + realistic seated pricing model

### Issue 1 diagnosis: "票价 80 起，场内最低 50"

Root cause is a **V1 seed inconsistency**, surfaced by the old price-display logic:

- `V1__baseline_schema_and_seed.sql` generates ONE shared seat pool for **every** SEATED schedule (rows 1-3 = 150, 4-7 = 100, 8-10 = **50**), regardless of each schedule's `price_range` string.
- Swan Lake's `sch-201` therefore has seats from 50, but its `price_range` string says **'$80 - $200'** (copied from the Opera House tier sheet). The V2 schedule `sch-ballet-default` snapshots `lay-opera-seated` (80/120/200), which matches its string.
- The old frontend/backend displayed the `price_range` **string** → "From $80"; the seat map sells from 50.

No new code was needed for this beyond the uncommitted **PriceBounds** work (live min/max from `schedule_seat`), which already replaces the string with the real minimum — **it requires a backend restart to take effect**. After the V6 migration below, the seed pools are re-tiered but economy remains the same value, so Swan Lake correctly shows "from 50" (sch-201 pool) / real bounds per schedule.

### Issue 2: realistic seated pricing — front rows cheapest, middle premium, back standard

Previous generator put VIP in rows 1-30%, standard next, economy at the back — backwards versus real theaters. New row allocation (identical in both generators and the migration):

```
front_rows  = rows >= 3 ? max(1, round(rows * 0.2)) : 0   → economy (B)
premium_end = max(front_rows + 1, ceil(rows * 0.65))      → premium (VIP)
rest                                                       → standard (A)
```

- `service/AdminService.java` `generateSeatPool` — re-tiered; comment cross-references the layout generator.
- `service/VenueManagementService.java` `generateSeatedLayout` — same change for layout templates.
- **NEW `db/migration/V6__realistic_seated_pricing.sql`** — re-tiers EXISTING data with the same rule:
  - `seat_layout_seat` (pure seated rows, `area_id IS NULL`) per layout — so future snapshots inherit the new model;
  - `schedule_seat` (`area_id IS NULL`) per schedule, **skipping SOLD seats** (order/ticket amounts must not drift) and CANCELLED/ENDED schedules;
  - the three tier prices are derived per layout/schedule from existing seat prices (max = VIP, min = economy, second-highest = standard; 2-tier/1-tier pools degrade gracefully);
  - MIXED balcony seats (area-priced, `area_id` set) are untouched.
- `test/.../VenueManagementServiceTest.java` — NEW `createSeatedLayoutPricesFrontCheapMiddlePremium` (10×4 layout: rows 1-2 = B/50, 3-7 = VIP/150, 8-10 = A/100); added the `ArgumentCaptor` import.
- Frontend needs no change: the 3D preview colors by status and positions by row/col; seat-map tier colors are assigned by price, not by section name.

Note: SOLD seats keep their original section/price by design, so a schedule with prior sales can show a few legacy-priced seats amid the re-tiered pool — acceptable for the course demo; amounts stay consistent with paid orders.
