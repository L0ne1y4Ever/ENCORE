## Round 2 — End-to-end realtime sync overhaul + defect cleanup (layered on top of the still-uncommitted venue/layout work; NOT yet committed)

This section records a second pass made on top of the venue/hall/layout/schedule work described above. That earlier work was still uncommitted when this round started, so there is no separate commit boundary — **this file-by-file list is the only way to tell Round-2 edits apart from the prior uncommitted work.** None of it has been committed; the user will commit everything together.

### Why

1. ZONED/MIXED **area inventory changes were never broadcast** over WebSocket, so users and admins saw stale availability for anything that was not a per-seat `schedule_seat` change. Pure-ZONED seat-selection did not even subscribe.
2. MIXED **double-counted seated balcony areas**: those areas have both a `schedule_area_inventory` row and `schedule_seat` rows, but seat sales only update `schedule_seat`, so inventory stayed at full capacity. `AdminService` summed the stale inventory (inflated totals) while `ShowService` counted correctly — the two disagreed.
3. A batch of smaller real defects and dead code identified during review.

## Round 3 — Auth gateway / HTTP status semantics / consistency hardening / structured seat label / bundle splitting / slice tests (layered on the still-uncommitted Round 1+2 work; NOT yet committed)

Third pass layered on top of the venue/layout (Round 1) and realtime-sync/defect-cleanup (Round 2) work above. All of it is still uncommitted, so — as with Round 2 — **this file-by-file list is the only way to tell Round-3 edits apart from the prior uncommitted work.** The user will commit everything together.

### Why

The post-Round-2 review surfaced eight remaining issues, grouped into four themes:

1. **Auth was scattered and used the wrong status code.** There was no HTTP-layer auth gate; every protected endpoint relied on hand-written `ensureAdminRole`/`ensureCheckInRole` calls in the service layer, so a single missing guard meant an unprotected route. Sa-Token had no `StpInterface` role source and no `NotRoleException` handler, so a wrong-role request returned 401 instead of 403, and every `BusinessException` returned HTTP 400 regardless of its semantic code (404/409/...).
2. **Redis locks could leak on transaction rollback.** Seat locks are written to Redis (not transactional); if the order DB transaction rolled back after the lock write, the lock survived pointing at a non-existent order id.
3. **The English e-ticket still showed a hardcoded Chinese seat label.** The label was a server-formatted `第 X 排 Y 号` string with no structured row/col for the client to localize.
4. **Build/test/i18n loose ends.** A 1.17 MB monolithic `index` chunk, two leftover hardcoded `city: '上海'` admin form defaults, and no slice tests for the new auth/error-code behavior.

### Backend — #6 HTTP status semantics

- `common/ErrorCode.java` — added `FORBIDDEN = 403` (the constant values already double as HTTP statuses: 400/401/403/404/409/500).
- `exception/GlobalExceptionHandler.java` — `handleBusinessException` now returns `ResponseEntity<ApiResponse<Void>>` with status = `HttpStatus.resolve(ex.getCode())` (fallback 400), so a `BusinessException(NOT_FOUND, ...)` produces a real HTTP 404 with the same body code. The body shape (`{code,msg,data}`) is unchanged, so the frontend (which reads `body.code`/`body.msg` and has no 4xx interceptor) is unaffected.

### Backend — #5 unified Sa-Token auth interceptor (HTTP-layer gate)

- **NEW `satoken/StpInterfaceImpl.java implements StpInterface`** — role source: `getRoleList(loginId)` looks up `user_account` by id and returns its single `role` (empty list when the user or role is missing); `getPermissionList` returns empty.
- **NEW `config/SaTokenConfigure.java implements WebMvcConfigurer`** — registers a `SaInterceptor` over `/**` and explicitly checks only protected prefixes:
  - login required: `/api/orders/**`, `/api/group-orders/**`, `/api/auth/me`, `POST /api/schedules/*/seats/lock`
  - role required: `/api/admin/**` → `admin`/`sysadmin`; `/api/checkin/**` → `checker`/`admin`/`sysadmin`
  - everything else (health, `auth/login|register|logout`, `/api/shows/**`, `GET` seats/areas, knife4j, `/ws`) matches no rule and passes through.
- `exception/GlobalExceptionHandler.java` — added a combined `NotRoleException` + `NotPermissionException` handler → HTTP 403 / body 403; the existing `NotLoginException` handler returns 401.
- Service-layer `ensureAdminRole`/`ensureCheckInRole` are **kept** as defense-in-depth (and existing tests depend on them); the interceptor is the primary HTTP gate, closing the "one missing guard = bare route" hole. Note: login *failure* now surfaces as HTTP 401 (was 400); the frontend toast text is unchanged because it reads `body.msg`.

### Backend — #1 seated-area inventory consistency hardening (no row deletion)

- `service/VenueManagementService.java`
  - `snapshotLayoutToSchedule` — for `isSeated` areas the inventory row's `total`/`available` are now initialized from the actual generated seat count for that area (not the raw `capacity`), with a comment stating that `schedule_seat` is the count source of truth and the inventory row is only the area registry / price / enumeration / stadium-map tile.
  - `updateAreaInventory` — now rejects edits to a seated area with `BusinessException(CONFLICT, "座位区库存请在座位快照中按座位管理")`, since seated-area counts are derived from `schedule_seat` and a bulk inventory edit would be read-side meaningless.
- `service/AdminService.java` `createSchedule` (ZONED/MIXED branch) — added the same contract comment; the seated-area loop already generates exactly `capacity` seats, so `total/available = capacity` is already seat-consistent here.
- **Deliberately not done:** deleting seated-area inventory rows — `SeatService.listScheduleAreas` enumerates areas *from* `schedule_area_inventory`, so removing those rows would make seated balconies vanish from the stadium map and become unreachable for seat selection.

### Backend — #2 Redis lock / transaction-rollback consistency

- `service/OrderService.java`
  - `createOrder` (SEATED) — after `attachOrderToLocks`, registers a `TransactionSynchronization`; on `afterCompletion(STATUS_ROLLED_BACK)` it calls `seatService.releaseLocks(scheduleId, seatIds)`, so a rolled-back order frees its seats immediately instead of leaving an orphan lock pointing at a non-existent order id.
  - `createGroupOrder` — after `transferLocksOwner(... lockOwner → orderId ...)`, registers a rollback callback that reverts the locks back to `lockOwner` (`transferLocksOwner(... orderId → lockOwner ...)`, `SeatService.SEAT_LOCK_TTL`), preserving the group session's seats.
  - Both helpers (`releaseSeatLocksOnRollback` / `revertGroupLocksOnRollback`) are guarded by `TransactionSynchronizationManager.isSynchronizationActive()` (inert under plain Mockito unit tests) and swallow callback-time `RuntimeException`s (any residual lock is reaped by the existing `cleanupExpiredSeatLocks` scheduler).

### Backend + frontend — #4 structured seat label

- `dto/TicketItemResponse.java` — added `Integer rowNo, Integer colNo` (the server-formatted `seatLabel` stays as a fallback).
- `service/OrderService.java` `toOrderResponse` — seat tickets set `rowNo/colNo` from the seat; area tickets leave them null (single construction site).
- `mock/orders.ts` — `TicketItem` gained `rowNo?`/`colNo?`.
- `views/user/ETicket.vue` + `views/user/Profile.vue` — seat display now uses `t('seat.info', { row, col })` (zh `第 {row} 排 {col} 号` / en `Row {row} Seat {col}`), falling back to `seatLabel || seatId`, so the English e-ticket no longer shows a hardcoded Chinese label.

### Frontend — #9 bundle splitting

- `vite.config.ts` — added `build.rollupOptions.output.manualChunks` splitting `three`, `echarts`+`zrender`+`vue-echarts`, and `element-plus`+`@element-plus` into separate vendor chunks, and raised `chunkSizeWarningLimit` to 1200.

### Frontend — #11 admin i18n cleanup

- `views/admin/Venues.vue` — removed the two hardcoded `city: '上海'` form defaults (now empty). The remaining admin views were already fully `t()`-driven.

### Backend — #8 infra-free slice tests

- **NEW `test/.../satoken/StpInterfaceImplTest.java`** — 4 Mockito tests: role returned for an existing account, empty for a missing user, empty when the role is null, and `getPermissionList` always empty.
- **NEW `test/.../web/GlobalExceptionHandlerTest.java`** — 4 standalone-MockMvc tests (`standaloneSetup(probeController).setControllerAdvice(new GlobalExceptionHandler())`) asserting `BusinessException(400/401/404/409)` maps to the matching HTTP status and body code. No DB/Redis/Sa-Token needed.
- The interceptor's pass/401/403 gating is left to the curl matrix below (a clean slice can't drive Sa-Token routing plus the Redis-backed role dao).
