## 关键实现

- 时间窗口为开演前 2 小时到演出结束，直接使用现有 `show_schedule.start_time` / `end_time`，不新增数据库字段。
- 检票服务在订单支付状态和票据状态通过后，再读取场次并执行窗口校验；失败时不会更新 `ticket_item`。
- 后端错误仍使用中文业务消息，由现有前端错误处理直接显示到检票页。
- 时间窗口轮最初没有修改 `CheckInRequest` / `CheckInResponse`；当前场次绑定轮已在 `CheckInRequest` 中新增可选 `scheduleId`，`CheckInResponse` 保持不变。
- 浏览器证据使用真实后端链路，订单和票据保留在本地数据库作为演示数据。
- WebSocket 只负责广播展示态，锁座、下单、支付等写操作仍全部走现有受保护 HTTP API。
- 实时事件基于最终业务状态发布；支付成功只广播 `SOLD`，不会因为 Redis 解锁动作误报 `AVAILABLE`。
- Dashboard WebSocket 只发送轻量刷新事件，不传聚合指标；页面收到事件后复用现有 `/api/admin/dashboard` 拉取真实指标。
- 检票台当前场次绑定只保存在浏览器本地，不新增检票台实体或数据库表；后台强制核销继续作为运营纠错通道。
- Three.js 仅在前端选座页使用；浏览器不支持 WebGL 时显示 fallback 文案，不阻塞下方 2D 选座和 HTTP 锁座流程。
- 3D 预览保持无框剧场带状区域，不替代 2D 座位图，避免影响移动端和精确选座。

## 问题与解决

- 问题：如果直接使用 `LocalDateTime.now()`，时间窗口单测会随系统时间漂移。
- 解决：通过 Spring 注入 `Clock`，单测使用固定时间源验证窗口边界和拒绝路径。
- 问题：当前检票接口只传票码，没有检票台当前场次上下文。
- 解决：先按已确认范围完成时间、取消和缺失场次校验；随后新增可选 `scheduleId` 和检票页当前场次绑定，保持旧调用兼容。
- 问题：如果把 Redis 解锁动作直接映射为“可售”，支付完成时会先把座位错误广播为 `AVAILABLE`。
- 解决：将事件发布放在订单/后台业务结果处，由最终状态决定广播内容；底层 `releaseLocks()` 保持静默。
- 问题：启动验证时 8080 仍被旧 Java 进程占用，真实 STOMP 脚本一开始连到了旧后端。
- 解决：通过 `netstat` 定位旧 PID，停止旧后端和旧 Vite 进程后，用当前源码重启服务再验证。
